# 项目文件修改规范

在修改现有项目文件前，必须遵循以下流程以避免破坏已有功能。

## 核心原则

**只读优先**：修改前先读取，理解后再动手

## 修改前检查流程

### 1. 文件存在性检查

```bash
# 确认文件是否存在
ls -la <目标文件路径>

# 搜索相关文件
find . -name "*.java" | grep -i <关键词>
```

### 2. 文件内容分析

读取文件时关注：
- 包结构和类名
- 字段定义和方法签名
- 依赖的其他类和接口
- 现有功能的用途

### 3. 影响范围评估

修改前检查：
- 该文件被哪些其他文件引用
- 修改会影响哪些下游功能
- 是否有测试依赖此文件

## 修改中规范

### 4. 最小化修改原则

- 只修改必要的部分
- 保留原有注释和文档
- 避免重命名无关的变量或方法
- 保持代码风格一致

### 5. 增量式修改

- 每次只做一个逻辑修改
- 修改后立即验证（编译/测试）
- 避免一次性重写整个文件

### 6. 备份策略

重要修改前：
```bash
# 创建备份
cp File.java File.java.bak

# 或使用 git
git stash  # 暂存当前修改
```

## 修改后验证

### 7. 编译检查

```bash
# 快速编译检查
mvn compile -q

# 完整构建
mvn clean install -DskipTests
```

### 8. 功能验证

- 运行相关测试
- 检查日志输出
- 验证业务逻辑

## 典型错误案例

### 案例1：覆写同名文件

```
问题：新项目中存在 EdgeAnomalyProcessingJob.java
      模型直接创建新文件覆盖了原有实现
```

正确做法：
```bash
# 1. 检查是否已存在
ls -la src/edge-processing/src/main/java/**/EdgeAnomalyProcessingJob.java

# 2. 读取现有文件内容
cat <文件路径>

# 3. 决定是修改还是新建
# 如果需要保留原有功能 → 修改现有文件
# 如果是完全不同的实现 → 使用不同文件名
```

### 案例2：修改依赖文件

```
问题：修改了 pom.xml 但没有检查其他模块的依赖关系
```

正确做法：
```bash
# 1. 检查依赖关系
mvn dependency:tree

# 2. 检查哪些模块依赖此文件
grep -r "artifactId" */pom.xml

# 3. 验证修改后其他模块仍能编译
mvn compile -pl <依赖模块>
```

## 检查清单

修改文件前，回答以下问题：

- [ ] 我是否读取了现有文件内容？
- [ ] 我是否理解了文件的现有功能？
- [ ] 我是否检查了该文件被哪些其他文件引用？
- [ ] 我的修改是否会影响其他模块？
- [ ] 修改后我是否运行了编译检查？
- [ ] 修改后我是否运行了相关测试？

## 常见场景处理

| 场景 | 处理方式 |
|------|---------|
| 文件已存在，需要相同功能 | 读取现有实现，评估是否需要修改 |
| 文件已存在，需要不同功能 | 使用新文件名，避免冲突 |
| 依赖版本冲突 | 检查 pom.xml，使用兼容版本 |
| API 不兼容 | 查阅框架文档，使用兼容方式 |

## 工具使用建议

```bash
# 搜索文件
find . -name "*.java" -type f

# 搜索类定义
grep -r "class <ClassName>" --include="*.java"

# 搜索方法调用
grep -r "<methodName>" --include="*.java"

# 检查依赖
mvn dependency:analyze
```
